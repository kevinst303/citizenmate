# Phase 08-05: Advanced Blog CMS Enhancement

**Status:** Planned  
**Depends on:** Phase 08-04 (Admin Dashboard UI Refinement)  
**Date:** 2026-05-06

---

## Goal

Enhance the admin blog management with WYSIWYG editor, feature images, tags, multi-language content, SEO optimization, and AI writing via DeepSeek. Unify the currently disconnected admin (Supabase) and frontend (MDX files) into a single database-driven blog system.

---

## Audit Summary

| Finding | Detail |
|---------|--------|
| **Disconnect** | Admin writes to Supabase `blog_posts` (flat schema); frontend reads from `src/content/blog/*.mdx` — posts created in admin never appear on frontend |
| **Rich Schema** | `public_schema.sql` already defines `blog_posts` + `blog_translations` + `blog_media` + `locales` + `get_blog_post_by_slug()` RPC — but was never implemented in app code |
| **Current Editor** | Plain `<textarea>` with "HTML supported" placeholder — no formatting, no image upload |
| **No Tags** | No tags table, no tag UI |
| **No Featured Images** | Images are static files in `/public/images/blog/` referenced by MDX frontmatter only |
| **No i18n Content** | 5 locales exist but blog content is English-only |
| **No SEO** | SEO fields exist in `blog_translations` schema but are never populated |

---

## Decisions

| ID | Area | Decision |
|----|------|----------|
| D-01 | Data Architecture | Full rich schema: `blog_posts` + `blog_translations` (i18n) + `blog_media` (images) + `blog_tags`. Admin writes to Supabase, frontend reads from Supabase. Migrate existing MDX posts to DB. |
| D-02 | WYSIWYG Editor | Tiptap v3 — free/MIT, headless, extensible, outputs semantic HTML compatible with Tailwind Typography prose |
| D-03 | AI Integration | Inline AI assistant in editor sidebar — generate/rewrite/translate content, suggest SEO fields and tags via DeepSeek API proxy routes |
| D-04 | i18n Workflow | Per-locale tabs in blog editor (EN \| ES \| HI \| ZH \| AR). Shared fields: tags, featured image, status. Per-locale: title, slug, content, SEO. |
| D-05 | Image Management | Upload to Supabase Storage `blog-images` bucket. Drag-drop/paste in Tiptap. 5MB limit, JPEG/PNG/WebP/GIF. |
| D-06 | SEO Scope | Per-locale SEO fields + JSON-LD structured data + sitemap integration + AI suggest |
| D-07 | Tags | Free-text input with autocomplete from existing tags. `blog_tags` + `blog_post_tags` junction table. |
| D-08 | Existing Content | Migrate all 25 MDX posts to Supabase with EN locale. Feature images referenced by URL in `blog_media`. |
| D-09 | AI Model | DeepSeek `deepseek-v4-flash`, non-thinking mode, temperature 0.7 for writing, 0.3 for translation. OpenAI SDK compatible. |
| D-10 | API Key Security | DeepSeek key stored in `.env.local`, served via Next.js API route proxy — never exposed to client |

## Database Changes

### Migration: `20260507000000_blog_tags.sql`

```sql
CREATE TABLE IF NOT EXISTS public.blog_tags (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(50) NOT NULL,
    slug VARCHAR(50) NOT NULL UNIQUE,
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE IF NOT EXISTS public.blog_post_tags (
    blog_post_id UUID NOT NULL REFERENCES public.blog_posts(id) ON DELETE CASCADE,
    tag_id UUID NOT NULL REFERENCES public.blog_tags(id) ON DELETE CASCADE,
    PRIMARY KEY (blog_post_id, tag_id)
);

-- RLS: public read
ALTER TABLE public.blog_tags ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.blog_post_tags ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Public can read blog tags" ON public.blog_tags FOR SELECT USING (true);
CREATE POLICY "Public can read blog_post_tags" ON public.blog_post_tags FOR SELECT USING (true);
CREATE POLICY "Admins can manage blog tags" ON public.blog_tags
    USING (public.is_admin()) WITH CHECK (public.is_admin());
CREATE POLICY "Admins can manage blog_post_tags" ON public.blog_post_tags
    USING (public.is_admin()) WITH CHECK (public.is_admin());
```

### Update existing migration: Add blog_posts columns

The current migration `20260501000000_super_admin.sql` creates a flat `blog_posts` table. The production `public_schema.sql` has the richer schema with `status` VARCHAR and separate `blog_translations`. We need to alter the existing table to match:

```sql
-- Alter existing blog_posts table to add missing columns
ALTER TABLE public.blog_posts 
  ADD COLUMN IF NOT EXISTS status VARCHAR(20) DEFAULT 'draft',
  ADD COLUMN IF NOT EXISTS is_featured BOOLEAN DEFAULT FALSE,
  ADD COLUMN IF NOT EXISTS reading_time INTEGER,
  ADD COLUMN IF NOT EXISTS sort_order INTEGER DEFAULT 0;
```

## Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| Existing flat `blog_posts` table conflicts with rich schema | Medium | Migrate existing data, alter table to add missing columns |
| Tiptap HTML output not matching Tailwind Typography prose | Low | Prose plugin styles raw HTML tags natively; test with existing content |
| DeepSeek API rate limits or downtime | Low | Retry logic with exponential backoff; AI is optional enhancement |
| Admin form state complexity with 5 locales | Medium | Use controlled form state with locale index; save per-tab |
| Migration script data loss | Medium | Back up production DB before running migration; idempotent design |
