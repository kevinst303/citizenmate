# Plan 07-01: Internationalization (i18n)

<objective>
Implement internationalization (i18n) support for 4 new languages (Spanish, Hindi, Mandarin, Arabic) in the application UI and core questions.
</objective>

## Tasks

1. **i18n Configuration and Middleware**
   <read_first>
   - src/middleware.ts (to see current routing and auth logic)
   - src/i18n/config.ts (if exists, to establish supported locales)
   </read_first>
   <action>
   Create `src/i18n/config.ts` if it doesn't exist, defining `locales: ['en', 'es', 'hi', 'zh', 'ar']` and `defaultLocale: 'en'`.
   Update `src/middleware.ts` to implement language negotiation and routing redirection if the locale is missing from the URL. Preserve existing auth routing logic.
   </action>
   <acceptance_criteria>
   - `src/i18n/config.ts` exports the correct locale arrays.
   - `src/middleware.ts` correctly rewrites/redirects to `/[lang]/...` when a language is set or negotiated.
   </acceptance_criteria>

2. **Translation Dictionaries**
   <read_first>
   - src/i18n/config.ts
   </read_first>
   <action>
   Create JSON dictionary files in `src/i18n/dictionaries/`: `en.json`, `es.json`, `hi.json`, `zh.json`, `ar.json`.
   Add base UI translation keys (e.g., `navigation.home`, `dashboard.title`, `upgrade.button`).
   Implement a `getDictionary` function in `src/i18n/getDictionary.ts` to load these JSON files asynchronously.
   </action>
   <acceptance_criteria>
   - JSON files exist for all 5 languages in `src/i18n/dictionaries/`.
   - `getDictionary.ts` exports an async function that returns the parsed JSON for the requested locale.
   </acceptance_criteria>

## Verification
- Navigate to `/es` and confirm the UI renders Spanish text.
- Verify that standard English routing `/` redirects to `/en` or sets default locale correctly based on the middleware implementation.
